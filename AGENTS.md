# AGENTS.md

## Project Overview

TeamFlow is an internal project management platform for engineering teams of 9–15 people. It provides work item management, sprint planning, kanban boards, release tracking, and team collaboration, with a real-time frontend backed by a .NET 10 API.

## Tech Stack

- **API:** .NET 10 / C# 14 / ASP.NET Core Controller-based / Clean Architecture + Vertical Slice
- **Frontend:** Next.js 16 (App Router) / React 19 / TypeScript / TanStack Query / Zustand
- **Database:** PostgreSQL 17 via EF Core + Npgsql
- **Messaging:** RabbitMQ 4 via MassTransit
- **Realtime:** SignalR (`@microsoft/signalr` v8)
- **Background Jobs:** .NET Hosted Services + Quartz.NET
- **Auth:** JWT (HMAC-SHA256, 30-min expiry) + Refresh Tokens (SHA-256 hashed, BCrypt passwords)
- **Testing:** xUnit + FluentAssertions + NSubstitute + Testcontainers (real PostgreSQL)
- **Frontend E2E:** Playwright

## Coding Rules

- Controllers call `Sender.Send()` only — no direct service injection in controllers
- All handlers return `Result<T>` or `Result` (CSharpFunctionalExtensions) — no throwing for business errors
- No business logic in Infrastructure — only persistence and external service calls
- No business logic in Controllers — only HTTP mapping
- Each feature in its own slice folder — no cross-slice handler imports
- Permission check first in every command handler that modifies data — use `IPermissionChecker`
- `WorkItemHistories` is append-only — never UPDATE or DELETE this table
- All API errors return `ProblemDetails` (RFC 7807) — never plain strings
- All new classes must be `sealed` by default unless designed for inheritance
- Test-First Development (TFD) — write failing tests before implementation; no exceptions
- Use `[Theory]` + `[InlineData]` to avoid separate `[Fact]` per edge case
- Never write permission resolution logic — use `IPermissionChecker`
- Never write JWT generation/validation without human review — `AuthService` is human-reviewed
- Never write to `work_item_histories` directly — always use `IHistoryService`
- Never log passwords, tokens, or PII
- Never commit secrets to source control
- Never DROP a database column in the same deploy — always backward-compatible migrations

## Team Learnings

- `ICurrentUser` is resolved from JWT claims in the HTTP context — Phase 1 used a stub; Phase 2+ uses real data
- `SignalRBroadcastConsumer` runs in `BackgroundServices`, not Api — it uses `IHubContext<TeamFlowHub>` to broadcast
- `BurndownSnapshotJob` broadcasts `burndown.updated` directly to the `sprint:{sprintId}` SignalR group after each daily write
- Refresh tokens are stored as SHA-256 hashes — never store the raw token in the database
- `EventPartitionCreatorJob` must run before the end of each month to pre-create the next month's `domain_events` partition

### Domain Context

**Feature areas implemented (Phases 0–3):**
- Auth: register, login, refresh token, change password, logout
- Organizations: create, get, list
- Teams: full CRUD + member management (add, remove, change role)
- Project Memberships: add/remove members (user or team), list, get current user's permissions
- Projects: full CRUD + archive
- Work Items: full CRUD, hierarchy (Epic → UserStory → Task/Bug/Spike), status transitions, assign/unassign, move, soft delete with cascade
- Work Item Linking: 6 link types, circular dependency detection, blockers check
- Work Item History: paginated append-only audit trail
- Backlog: paginated with full filter matrix + full-text search + reorder
- Kanban: board grouped by status columns (ToDo / InProgress / InReview / Done)
- Releases: full CRUD + assign/unassign work items
- Sprints: full CRUD, start/complete lifecycle, add/remove items, per-member capacity, burndown chart data

**Background jobs:**
- `BurndownSnapshotJob` — daily per active sprint; detects At Risk (remaining > ideal × 1.2)
- `ReleaseOverdueDetectorJob` — daily; flags overdue releases with incomplete items
- `StaleItemDetectorJob` — daily; flags work items with no recent updates
- `EventPartitionCreatorJob` — monthly; pre-creates `domain_events` partitions

**Key enums:**

- `WorkItemStatus`: ToDo, InProgress, InReview, NeedsClarification, Done, Rejected

## How to Build

**Prerequisites:** .NET 10 SDK, Node.js 20+, Docker (for local infrastructure)

### Backend

```bash
# Start infrastructure (PostgreSQL + RabbitMQ)
docker compose up postgres rabbitmq -d

# Build solution
dotnet build TeamFlow.slnx

# Apply EF Core migrations
dotnet ef database update --project src/core/TeamFlow.Infrastructure --startup-project src/apps/TeamFlow.Api

# Run API
dotnet run --project src/apps/TeamFlow.Api

# Run BackgroundServices
dotnet run --project src/apps/TeamFlow.BackgroundServices
```

### Frontend

```bash
cd src/apps/teamflow-web
npm install
npm run dev       # dev server (http://localhost:3000)
npm run build     # production build
```

### Docker (full stack)

```bash
# Copy and configure environment variables
cp .env.example .env

docker compose up --build
# API: http://localhost:5000
# RabbitMQ management: http://localhost:15672
```

dotnet test tests/TeamFlow.Application.Tests
