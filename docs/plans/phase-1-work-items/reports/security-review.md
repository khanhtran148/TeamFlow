### Summary

The Phase 1 codebase has two classes of security concern that must be resolved before any production deployment: (1) credentials and a JWT signing secret are committed to VCS in `appsettings.json` and `docker-compose.yml`, creating a permanent exposure risk; and (2) four Project mutation handlers (`Create`, `Update`, `Delete`, `Archive`) silently omit permission checks entirely, meaning those operations are effectively unprotected even when real RBAC is introduced in Phase 2. The auth and permission stubs are correctly documented and scoped, but the missing `[Authorize]` attributes on all controllers and the unvalidated SignalR group-join methods represent structural gaps that must be retrofitted before real JWTs are issued.

### Unresolved Questions

- Is there a `.gitignore` or pre-commit hook preventing `appsettings.json` (with the `Jwt:Secret` populated) from being committed in CI? If not, secret scanning should be added.
- What is the intended `Permission` enum value for `Project_Create`/`Project_Delete`/`Project_Archive`? The domain `Permission` enum needs these values confirmed before the missing handler guards can be added.
- Will Phase 2 introduce a `[Authorize]` policy on `ApiControllerBase`, or will each controller declare it individually? The approach should be decided now to avoid per-controller gaps.
